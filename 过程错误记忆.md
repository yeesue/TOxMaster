## 一、C++标准版本兼容性问题

### 发生时间
2026-03-29

### 任务
任务1 - 修复RAT_FUNC除零问题

### 严重程度
🔴 高 - 编译失败

### 错误描述
使用了C++17特性的 `<optional>` 头文件和 `[[deprecated]]` 属性，导致在Qt 5.9.9 / MinGW 5.3.0环境下编译失败。

```
xcp/ratfunc_converter.h:29:20: fatal error: optional: No such file or directory
```

### 错误原因
1. **环境认知不足**: 未提前确认目标编译环境的C++标准版本支持
2. **盲目使用新特性**: 直接使用了C++17特性，未考虑向后兼容
3. **验证流程缺失**: 编写代码后未先进行编译验证

### 环境约束

| 环境 | 版本 | C++标准支持 |
|-----|-----|------------|
| Qt | 5.9.9 | C++11为主 |
| MinGW | 5.3.0 | 部分C++14 |
| 配置 | `CONFIG += c++11` | C++11 |

### 解决方案

**替代 `std::optional`**：创建自定义结果结构体

```cpp
// 不兼容的代码
std::optional<qreal> toPhys(qreal intValue) const;

// 兼容的代码
struct RatFuncResult {
    bool success;
    qreal value;
    bool hasValue() const { return success; }
    qreal getValue() const { return value; }
};
RatFuncResult toPhys(qreal intValue) const;
```

**移除 `[[deprecated]]`**：改用注释说明

```cpp
// 不兼容的代码
[[deprecated("Use safeXxx instead")]]
qreal oldFunction();

// 兼容的代码
// 旧版函数（保留向后兼容，建议使用safe版本）
qreal oldFunction();
```

### 经验教训

1. **先确认环境再编码**: 开始任务前检查 `.pro` 文件中的 `CONFIG` 设置
2. **使用最低兼容标准**: 代码应满足项目配置的最低标准要求
3. **增量编译验证**: 每完成一个文件修改就尝试编译

### 后续预防措施

所有新代码必须兼容C++11标准。避免使用的C++14/17特性：

| C++14/17特性 | 替代方案 |
|-------------|---------|
| `std::optional` | 自定义结构体或 `QPair<bool, T>` |
| `std::variant` | 联合体或枚举+成员变量 |
| `[[deprecated]]` | 注释说明 |
| 结构化绑定 `auto [a, b] = ...` | 手动解包 |
| `if constexpr` | 模板特化或运行时判断 |
| `std::string_view` | `QString` 或 `const char*` |

## 二、Signal/Slot编译错误

### 发生时间
2026-03-31

### 任务
第四阶段 - 信号槽升级

### 严重程度
🔴 高 - 编译失败

### 错误描述
信号槽参数类型不匹配导致编译失败，主要体现在：
1. 重载信号未正确使用 `QOverload`
2. 信号和槽的参数类型不完全匹配
3. 第三方库类名拼写错误

### 典型错误案例

#### 案例1：重载信号处理
```cpp
// 错误：无法确定重载版本
connect(comboBox, &QComboBox::currentIndexChanged, this, &MyClass::onChanged);

// 正确：使用QOverload指定版本
connect(comboBox, QOverload<int>::of(&QComboBox::currentIndexChanged), 
        this, &MyClass::onIndexChanged);
```

#### 案例2：类名拼写错误
```cpp
// 错误：A2L_ProjectWin 类不存在
connect(a2lWin, &A2L_ProjectWin::winClosed, this, &SettingWin::projectWinClosedSlot);

// 正确：类名是 A2LProjectWin（无下划线）
connect(a2lWin, &A2LProjectWin::winClosed, this, &SettingWin::projectWinClosedSlot);
```

#### 案例3：QCPAxis重载信号
```cpp
// 错误：setRange 也是重载函数
connect(mPlot->xAxis, QOverload<const QCPRange&>::of(&QCPAxis::rangeChanged), 
        mPlot->xAxis2, &QCPAxis::setRange);

// 正确：槽函数也需要QOverload
connect(mPlot->xAxis, QOverload<const QCPRange&>::of(&QCPAxis::rangeChanged), 
        mPlot->xAxis2, QOverload<const QCPRange&>::of(&QCPAxis::setRange));
```

### 解决方案

对于复杂的重载信号槽连接，可以使用旧语法保持兼容：

```cpp
// 当新语法过于复杂时，使用旧语法
connect(sender, SIGNAL(valueChanged(int)), receiver, SLOT(onValueChanged(int)));
```

### 经验教训

1. **检查类名定义**: 使用 `&ClassName::` 时确保类名拼写正确
2. **重载信号处理**: 使用 `QOverload<参数类型>::of(&ClassName::signalName)`
3. **第三方库谨慎**: 第三方库可能使用旧语法，修改前需确认

## 三、A2ML解析崩溃问题

### 发生时间
2026-03-29

### 任务
任务5 - 添加A2ML解析器

### 严重程度
🔴 高 - 运行时崩溃

### 错误描述
解析A2ML解析器时，A2L文件解析崩溃。

### 错误原因
A2ML关键字被识别为 `Keyword` 类型而非 `Identifier` 类型，导致无法正确识别 `/end A2ML`。

### 解决方案

修改 `a2ml.cpp` 中的结束判断：

```cpp
// 修改前
if (token == Identifier && lex->getLexem() == "A2ML")

// 修改后
if ((token == Keyword || token == Identifier) && lexem == "A2ML")
```

### 经验教训

词法分析器可能将关键字识别为不同类型，需要同时检查多种可能的token类型。

## 五、常用Qt信号槽连接模式

### 普通连接
```cpp
connect(sender, &SenderClass::signalName, receiver, &ReceiverClass::slotName);
```

### 重载信号连接
```cpp
connect(comboBox, QOverload<int>::of(&QComboBox::currentIndexChanged), 
        receiver, &ReceiverClass::slotName);
connect(comboBox, QOverload<const QString&>::of(&QComboBox::currentIndexChanged), 
        receiver, &ReceiverClass::slotName);
```

### Lambda连接
```cpp
connect(button, &QPushButton::clicked, this, [this]() {
    this->doSomething();
});
```

### 带默认参数的连接
```cpp
// 使用旧语法处理默认参数问题
connect(sender, SIGNAL(signalWithDefault(int)), receiver, SLOT(slot(int)));
```
