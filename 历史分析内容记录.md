## 一、项目架构与功能分析

> 来源：代码分析.md、项目分析报告_最终版.md（两者内容高度重复，已合并去重）

### 1.1 项目概述

| 项目 | 主要功能 | 技术栈 |
|------|----------|--------|
| **XCP_COMM** | XCP协议标定测量上位机，支持多硬件CAN卡 | Qt5 Widgets, QCustomPlot, MDF3 |
| **A2L_Parser** | A2L文件解析器，支持多种标定文件格式 | Qt5, Quex词法生成器, OpenMP |

**数据流**: A2L文件 → 解析变量列表 → XCP通讯 → ECU标定/测量

### 1.9 早期识别的架构问题

| 问题 | 现状 | 影响 |
|------|------|------|
| 缺乏硬件抽象接口 | if-else硬编码判断硬件类型 | 扩展新硬件需修改多处代码 |
| 全局变量过多 | `globals.h`中大量静态变量 | 耦合度高，难以测试 |
| 线程管理复杂 | 每个功能创建多个线程 | 生命周期管理分散 |
| 错误处理不完整 | 缺少统一错误机制 | 问题定位困难 |
| 内存管理风险 | 手动new/delete | 运行时内存问题 |
| 性能瓶颈 | 大量使用QSharedMemory | 数据更新延迟，实时性差 |
| 界面响应慢 | UI线程与工作线程耦合 | 操作卡顿，用户体验差 |
| 功能扩展性差 | 硬编码的功能逻辑 | 难以添加新功能 |
| 配置管理混乱 | 多处配置文件，格式不统一 | 配置易丢失，管理困难 |
| 技术栈较旧 | Qt 5.9.9，C++11标准 | 无法使用现代C++特性 |

### 2.2 第一阶段：内存安全（P0-紧急）

**问题**：跨线程信号槽传递裸指针、线程异常终止时动态分配缓冲区无法释放、未使用智能指针。

**解决方案**：创建智能指针工具类 (`XCP_COMM/common/smart_ptr.h`)：

```cpp
using ByteArrayPtr = QSharedPointer<quint8>;
inline ByteArrayPtr makeByteArray(int size) {
    return ByteArrayPtr(new quint8[size], [](quint8* p) { delete[] p; });
}
using CharArrayPtr = QSharedPointer<char>;
inline CharArrayPtr makeCharArray(int size) {
    return CharArrayPtr(new char[size], [](char* p) { delete[] p; });
}
```

**信号槽修改**：`void RESDataReady(quint8 *data, quint32 len)` → `void RESDataReady(ByteArrayPtr data, quint32 len)`

### 2.3 第二阶段：线程优化（P0-紧急）

**问题**：`usleep(10)` 导致CPU占用过高、线程退出仅靠bool标志缺乏同步、线程生命周期管理不规范。

**解决方案**：引入条件变量机制 + 使用SDK超时替代忙等待：

```cpp
class XCP_R_Thread : public QThread {
    QMutex m_mutex;
    QWaitCondition m_condition;
    bool m_running = true;
};
void XCP_R_Thread::stop() {
    QMutexLocker locker(&m_mutex);
    m_running = false;
    m_condition.wakeAll();
}
```

## 三、分阶段重构计划

> 来源：重构计划.md、第一阶段落地任务清单.md（后者是前者阶段1的细化落地方案）

### 3.1 总体目标

- 解耦 UI / 业务流程 / 设备通信 / 数据缓存
- 收敛 XCP 新旧线程和驱动实现
- 收敛 QSharedMemory 与 MemoryManager 两套数据模型
- 建立可覆盖真实主链路的测试体系

### 3.2 阶段0：基线冻结与风险隔离

- 冻结当前可运行版本，建立 `codex/refactor-baseline` 分支
- 梳理主链路模块和遗留目录
- 整理编译入口和测试入口
- 不删除代码，只标记归属

### 3.3 阶段1：拆分MainWindow

**新增类**：`MainWindowContext`、`RecordService`、`ImportExportService`、`UiStatePresenter`

**RecordService 迁移内容**：
- `initMdfRecord()` / `endMdfRecord()` / 录制线程生命周期管理
- `on_actionRcdOn_triggered` / `on_actionRcdOff_triggered` / `on_actionRcdSet_triggered`
- 接口：`configure() / startRecording() / stopRecording() / isRecording() / setRecordRate() / setRecordFileName()`

**ImportExportService 迁移内容**：
- `exportMeasToExcel / exportCharsToExcel / exportXcpPamsToExcel`
- `importMeasFromExcel / importCharsFromExcel / getA2LPamsFromExcel`
- `getA2LPamsFromDB / getMeasPamsFromDB / getCharPamsFromDB`
- `saveMeasToDB / saveCalisToDB / delExcel / convertToColName`
- 先保留：文件选择对话框、菜单action入口、用户提示

**UiStatePresenter 迁移内容**：
- `setLED / sltXcpCanInitStatus / sltXcpConnStatus / sltXcpCaliStatus / sltXcpPollingStatus / sltXcpDaqRunStatus`

**暂不处理的成员**：`xcpMainThread / xcpMaster / xcpPollThread / charPamCheckThread / mapCharPamCheckThread / 双通道数据 / QSharedMemory链路`

**推荐执行顺序**：
1. 新增 `MainWindowContext` → 2. 新增 `RecordService` → 3. 迁移录制逻辑 → 4. 新增 `ImportExportService` → 5. 迁移Excel/DB/CSV逻辑 → 6. 新增 `UiStatePresenter` → 7. 收口状态显示 → 8. 瘦身MainWindow

### 3.4 阶段2：重构XCPMaster

**新增层次**：
- `XcpSession` — 会话管理
- `XcpCommandExecutor` — 命令执行
- `XcpDaqService` — DAQ服务
- `XcpCalibrationService` — 标定服务

**拆分顺序**：先抽 `XcpCommandExecutor` → 再抽 `XcpDaqService` → 再抽 `XcpCalibrationService` → `XCPMaster` 退化为 façade

### 3.5 阶段3：统一驱动与线程模型

- 统一走 `ICanDriver + XCP_Thread_Unified`
- 实施策略：新增统一路径 → 配置项支持 legacy/unified → 统一路径稳定后旧路径退化 → 最终删除旧线程
- 每种硬件至少做一次真实设备 smoke test

### 3.6 阶段4：统一共享数据模型

- 结束 `QSharedMemory` 与 `MemoryManager` 并存
- 建议替换为：`BufferHandle / BufferView / TypedBuffer<T> / DataBufferStore`
- 迁移顺序：先包住旧QSharedMemory → 统一外部接口 → 替换底层实现 → 移除裸QSharedMemory*

### 3.7 阶段5：清理A2L_Parser的DLL边界

- 将A2L解析内核从UI和DLL全局状态中解耦
- 拆分为：`A2L Core` + `A2L Query Service` + `A2L UI`
- 新接口：`A2lDocumentHandle* loadA2l(...)` / `A2lQueryResult getMeasurementInfo(...)` / `void freeDocument(...)`

### 3.8 各阶段验收标准

| 阶段 | 验收标准 |
|------|----------|
| 阶段1 | MainWindow体量下降，UI行为不变，录制与导入导出逻辑不再堆在窗口类 |
| 阶段2 | XCPMaster成为协调器，协议命令具备独立单测能力，DAQ与Calibration互不影响 |
| 阶段3 | 统一驱动链路可跑通至少一种硬件，支持legacy/unified切换 |
| 阶段4 | 业务层不直接操作裸QSharedMemory*，缓冲区访问统一入口 |
| 阶段5 | DLL不再暴露UI全局状态，解析接口独立于窗口 |

### 3.9 测试配套计划

- **协议测试**：CONNECT / GET_STATUS / SET_MTA / UPLOAD / DOWNLOAD / DAQ config

## 四、代码风格规范

> 来源：代码风格规范.md（完整保留，与代码审查流程.md中6.1/6.2节重复部分已去重）

### 4.1 基本规则

- **缩进**：4个空格，不使用制表符
- **命名**：类名 `XCPMaster`，函数名 `CTO_CMD_RES_Comm`，变量名 `xcpCanThread`，常量名 `ERR_SUCCESS`，成员变量建议 `m_` 前缀
- **大括号**：K&R风格，所有代码块必须使用大括号
- **空格**：运算符两侧、逗号后、分号后、冒号后、花括号前后各加一个空格
- **行长度**：不超过100字符

### 4.2 注释规范

- 类和函数：使用Doxygen风格
- 行注释：使用 `//`，与代码至少一个空格
- 块注释：使用 `/* */`，多行每行以 `*` 开头

### 4.3 代码结构

- **文件结构**：头文件包含 → 宏定义 → 类型定义 → 类定义 → 函数定义
- **类结构**：成员变量在前 → 构造/析构 → 公共函数 → 保护函数 → 私有函数
- **函数**：保持简短，只负责一个功能，参数不超过5个

### 4.4 C++特有规则

- 头文件保护：使用 `#ifndef`/`#define`/`#endif`，宏名 `XCP_ERROR_H`
- 包含顺序：系统头文件 → 第三方库 → 项目内部
- 智能指针：优先使用 `std::unique_ptr`/`std::shared_ptr`
- 异常处理：使用异常而非错误码，捕获时使用引用
- 命名空间：不在头文件中使用 `using namespace`

### 4.5 Qt特有规则

- 信号槽：命名清晰，参数匹配
- 内存管理：使用Qt父子机制，无父对象时使用智能指针
- 字符串：优先使用 `QString`，国际化使用 `tr()`

### 4.6 工具和自动化

- 代码格式化：clang-format
- 静态分析：Clang Static Analyzer
- 测试：Qt Test / Google Test

## 六、过程错误记忆

> 来源：过程错误记忆.md（完整保留，独有的历史经验记录）

### 6.1 C++标准版本兼容性问题

**发生时间**：2026-03-29 | **任务**：任务1 - 修复RAT_FUNC除零问题 | **严重程度**：🔴 高

**错误**：使用了C++17特性的 `<optional>` 和 `[[deprecated]]`，在Qt 5.9.9 / MinGW 5.3.0下编译失败。

**环境约束**：Qt 5.9.9（C++11为主）、MinGW 5.3.0（部分C++14）、`CONFIG += c++11`

**替代方案**：
- `std::optional` → 自定义 `RatFuncResult` 结构体或 `QPair<bool, T>`
- `[[deprecated]]` → 注释说明
- `std::variant` → 联合体或枚举+成员变量
- 结构化绑定 → 手动解包
- `if constexpr` → 模板特化或运行时判断
- `std::string_view` → `QString` 或 `const char*`

**教训**：先确认环境再编码，使用最低兼容标准，增量编译验证。

### 6.2 Signal/Slot编译错误

**发生时间**：2026-03-31 | **任务**：第四阶段 - 信号槽升级 | **严重程度**：🔴 高

**典型错误**：
1. 重载信号未使用 `QOverload`：`&QComboBox::currentIndexChanged` → `QOverload<int>::of(&QComboBox::currentIndexChanged)`
2. 类名拼写错误：`A2L_ProjectWin` → `A2LProjectWin`
3. QCPAxis重载：槽函数也需要 `QOverload`

**经验**：检查类名定义、重载信号使用QOverload、第三方库谨慎修改。

### 6.3 A2ML解析崩溃问题

**发生时间**：2026-03-29 | **任务**：任务5 | **严重程度**：🔴 高

**原因**：A2ML关键字被识别为 `Keyword` 而非 `Identifier`，导致无法正确识别 `/end A2ML`。

**修复**：`if (token == Identifier && lexem == "A2ML")` → `if ((token == Keyword || token == Identifier) && lexem == "A2ML")`

### 6.4 编码检查清单

**编码前**：
- 确认项目 `CONFIG` 设置（C++标准版本）
- 确认Qt版本和可用模块
- 确认编译器版本和支持特性
- 确认类名拼写正确

**编码时**：
- 不使用超出目标标准的语言特性
- Qt类型优先于STL类型
- 使用Qt容器保持一致性
- 重载信号使用 `QOverload`

**提交前**：
- 本地编译通过、无新增警告
- 代码风格与项目一致
- 功能测试通过

### 6.5 常用Qt信号槽连接模式

```cpp
// 普通连接
connect(sender, &SenderClass::signalName, receiver, &ReceiverClass::slotName);

// 重载信号连接
connect(comboBox, QOverload<int>::of(&QComboBox::currentIndexChanged), receiver, &ReceiverClass::slotName);

// Lambda连接
connect(button, &QPushButton::clicked, this, [this]() { this->doSomething(); });

// 带默认参数（使用旧语法）
connect(sender, SIGNAL(signalWithDefault(int)), receiver, SLOT(slot(int)));
```
