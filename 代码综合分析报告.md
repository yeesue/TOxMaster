## 一、项目概述

### 1.1 项目定位

TOxMaster 是一个基于 Qt 开发的**汽车电子标定测量工具**，核心功能是实现 XCP 协议通信与 A2L 文件解析。

```
┌─────────────────────────────────────────────────────────────────┐
│                         TOxMaster                                │
├───────────────────────────┬───────────────────────────────────┤
│      A2L_Parser            │         XCP_COMM                  │
│   (A2L 文件解析器)          │      (XCP 标定测量上位机)         │
├───────────────────────────┼───────────────────────────────────┤
│  • ASAP2/A2L 语法解析       │  • XCP 协议实现                   │
│  • HEX/DCM/SREC 解析       │  • CAN/CANFD 通讯                │
│  • 多格式数据导入导出        │  • DAQ 数据采集                   │
│  • 变量定义管理             │  • 标定/测量功能                  │
└───────────────────────────┴───────────────────────────────────┘
```

#### 4.2.2 内存泄漏深度分析

经过对项目代码的全面审查，识别出 **6大类、15+处** 潜在内存泄漏问题，按严重程度分级如下：

##### 🔴 严重级别（HIGH）— 必须修复

**问题 1：`Cali_Pair::data` 泄漏** — `charpamcheck.cpp`

`Cali_Pair` 结构体中的 `data` 字段使用裸 `new char[]` 分配，但只在标定成功时才 `delete`：

```cpp
// a2l/a2l_varchar.h:10-15
struct Cali_Pair{
    A2L_VarChar *charVar;
    char *data;       // ← 裸指针
    int size;
    int offset = 0;
};
```

- `charpamcheck.cpp:104` — `pair.data = new char[dataSize]` 分配内存
- `charpamcheck.cpp:290` — Map线程中同样 `pair.data = new char[dataSizeAG]`
- `xcp_polling_thread.cpp:74/97` — 只在 `XCP_Cal_VALUE` 成功时 `delete pair.data`
- **如果标定失败，`data` 永远不会被释放**
- **如果线程退出时 `caliPairList` 非空，所有排队的 `pair.data` 全部泄漏**

**修复建议**：将 `Cali_Pair::data` 改为 `CharArrayPtr` 智能指针（项目已有 `smart_ptr.h` 定义）。

---

**问题 2：`canDataForRecord` 信号裸指针泄漏** — `can_fun_thread.cpp`

```cpp
// can/can_fun_thread.cpp:42-48
quint8 *buf = new quint8[len-8];
memcpy(buf, data, 8);
memcpy(buf+8, data+16, len-16);
emit canDataForRecord(buf, len-8, keyName);  // ← 裸指针跨线程传递
```

`buf` 通过信号传递给 `MDF_Record_Thread::mdf_record_slot_raw`，虽然该函数在正常和异常路径都 `delete[] buf`，但存在以下风险：
- **如果信号槽连接使用 `Qt::QueuedConnection`（跨线程默认），`buf` 指针在到达槽之前可能因为对象析构而成为悬空指针**
- 如果目标对象已销毁，信号被丢弃导致 `buf` 永远泄漏

同样问题存在于：
- `can_zlg/can_thread_zlg.cpp:144,173,208,238` — ZLG CAN 线程
- `can_ts/can_thread_ts.cpp:216,251` — TS CAN 线程

**修复建议**：改用 `ByteArrayPtr` 智能指针传递信号数据。

---

**问题 3：`fromReadRawDataToMeasVars` 中 `temp` 泄漏** — `xcp_main_thread.cpp`

```cpp
// xcp_tester/xcp_main_thread.cpp:73
char *temp = new char[dataSize];
memcpy(temp, data+startByte, dataSize);
// ... 使用 temp 计算 measValue ...
// ← 从未 delete[] temp，每次循环泄漏一块内存
```

`temp` 在循环中每次 `new`，但在整个函数结束时 **从未 `delete[]`**。此函数每次调用为每个 `measPamList` 项分配一块内存，全部泄漏。

**修复建议**：使用栈数组或 `QByteArray` 替代，或在每次循环末尾 `delete[] temp`。

---

**问题 4：`loadDbc` / `loadDbcFromArxml` 中临时数组泄漏** — `can_fun_thread.cpp`

在 `loadDbc()` 函数中（同样在 `loadDbcFromArxml` 中重复），以下临时数组 **从未释放**：

| 变量 | 行号 | 泄漏原因 |
|------|------|----------|
| `clusterRef` | 518 | `new nxDatabaseRef_t[numOfClusterRef]`，无 `delete[]` |
| `clusterName` | 536 | `new char[sizeOfClusterName]`，无 `delete[]` |
| `frameRef` | 611 | `new nxDatabaseRef_t[numOfFrameRef]`，无 `delete[]` |
| `frameName` | 634 | `new char[sizeOfFrameName]`，无 `delete[]` |
| `sigRef` | 660 | `new nxDatabaseRef_t[numOfSigRef]`，无 `delete[]` |
| `sigName` | 676 | `new char[sizeOfSigName]`，无 `delete[]` |
| `sigUnit` | 677 | `new char[sizeOfSigUnit]`，无 `delete[]` |

这些在每次加载DBC数据库文件时都会泄漏，**且 `loadDbc` 和 `loadDbcFromArxml` 代码完全重复**，同样的问题出现了两遍。同样的问题也存在于 `cansetdlg.cpp` 和 `nixnet_thread.cpp` 的 `xnetDbcModify` 中。

**修复建议**：使用 `QByteArray` 或 `std::vector` 替代裸 `new`，使用后自动释放。

---

##### 🟡 中等级别（MEDIUM）— 建议修复

**问题 5：`QTimer` 泄漏** — `can_fun_thread.cpp:248`

```cpp
// can/can_fun_thread.cpp:248-252
QTimer *timer = new QTimer();
timer->setInterval(10);
connect(timer, SIGNAL(timeout()), this, SLOT(Slt_FrameSigValueUpdated()));
connect(timer, SIGNAL(timeout()), this, SLOT(Slt_WriteFrameSigValueUpdated()));
timer->start();
```

`timer` 没有 parent 也没有在 `stopCanComm()` 中 `delete`。每次调用 `startCanComm()` 都会创建一个新的 `QTimer`，旧的泄漏。

**修复建议**：设置 `this` 为 parent，或在 `stopCanComm()` 中显式删除。

---

**问题 6：`NIXNET_R_Queue_Thread::canfd_frameBuffer_Queue` 泄漏** — `nixnet_thread.cpp:74`

```cpp
// can/nixnet_thread.cpp:74
canfd_frameBuffer_Queue = new nxFrameCANFD_t[frameQueueSize];
```

读线程在 `run()` 中分配了 `canfd_frameBuffer_Queue`，但 `NIXNET_R_Queue_Thread::~NIXNET_R_Queue_Thread()` 是空的，**没有释放这块内存**。与 `NIXNET_W_Queue_Thread` 不同（写线程在 `run()` 结尾有 `delete[]`）。如果 `isStop=true` 导致循环退出，`canfd_frameBuffer_Queue` 不会被释放。

**修复建议**：在 `run()` 循环退出后添加 `delete[] canfd_frameBuffer_Queue`。

---

**问题 7：`fromCharVarsToWriteSM` 提前返回泄漏** — `xcp_main_thread.cpp:1015`

```cpp
// xcp_tester/xcp_main_thread.cpp:1015-1023
char *buffer = new char[sizeWrite];
if(!smWrite->isAttached())
{
    if(!smWrite->attach())
    {
        qDebug()<<"Unable attach to write sharedmemory.";
        return;  // ← buffer 泄漏
    }
}
```

如果 `attach()` 失败，`buffer` 未被释放。

**修复建议**：使用 `QByteArray` 或 RAII 守卫确保提前返回时释放内存。

---

**问题 8：`QSharedMemory` 与 `MemoryManager` 两套并存管理混乱**

项目中存在两套并存的内存管理体系（在 `mainwindow.h` 中可以看到注释掉的旧 `QSharedMemory` 指针），部分代码已迁移到 `MemoryManager`，部分仍使用裸 `QSharedMemory*`：

- `xcp_main_thread.h:169-186` — 仍然保留 `smRead`, `smWrite`, `smMapWriteList`, `smMapWriteHash` 等裸 `QSharedMemory*`
- `nixnet_thread.cpp:706,736` — `new QSharedMemory(this)` 传入 `this` 作为 parent，但对象生命周期管理不清晰
- `charpamcheck.cpp:47-49` — 栈上创建 `QSharedMemory` 但不检查 `attach()` 返回值

**修复建议**：按重构计划统一迁移到 `MemoryManager`，移除裸 `QSharedMemory*` 使用。

---

##### 🟢 低级别（LOW）— 改善建议

**问题 9：`pollDataForRecord` 双重 emit 导致双重释放或泄漏** — `xcp_polling_thread.cpp:163-164`

```cpp
// xcp/xcp_polling_thread.cpp:146-164
quint8 *buf = new quint8[smReadSize + 8];
copyFromSharedMemory(&sm, 0, (char*)buf, smReadSize + 8);
emit pollDataForRecord(buf, smReadSize + 8, -1);
emit pollDataForRecord(buf, smReadSize + 8, "RP_POLL_" + xcpName);
```

同一个 `buf` 被 emit 了两次，但 `mdf_record_slot_raw` 在使用后会 `delete[] buf`。如果两个连接都活跃，第二个 `delete[]` 是未定义行为；如果录制关闭，第一个槽就 `delete[]` 了，第二个槽访问已释放内存。

**修复建议**：分配两个独立 buffer 或使用智能指针。

---

**问题 10：`PARAM*` 对象生命周期管理混乱**

`fromDbcFrameSigsToPams()` 中创建的 `PARAM*` 对象被放入 `readPamListHash` / `writePamListHash`，但在 `stopCanComm()` 中：
- `readPamList` 和 `writePamList` 被 `qDeleteAll` 清理
- 但 `readPamListHash` 和 `writePamListHash` 只是 `.clear()`，**hash 中指向的 PARAM* 可能与 list 中的相同，也可能不同**，存在双重释放或泄漏风险

**修复建议**：统一所有权，使用智能指针管理 PARAM 对象。
